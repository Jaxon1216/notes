## 例题

- 推荐开区间的二分查找算法
  - 由于索引从0开始，偶数长度的时候视觉上的mid指着的元素通常会偏左
  - 循环不变量是我们始终约定的一套规则，遵循这个规则来，退出循环，推理答案
  - 记住这个例题
- 为什么循环条件是 `l + 1 < r`？
  在 (l, r) 这个区间内，当 `l + 1 === r` 时，区间内没有元素了，停止循环。
  所以循环继续的条件是 `l + 1 < r`，等价于 r - l > 1。
- 为什么第一个 if 是 `target <= nums[mid]`？
  因为我们要找的是第一个大于等于 target 的数的下标。
    - 如果 `nums[mid] >= target`，说明第一个大于等于 target 的数可能在 mid 或 mid 左边（包括 mid 本身），所以 r 应该收缩到 mid。
    - 如果 `nums[mid] < target`，说明 mid 以及它左边的元素都小于 target，第一个大于等于 target 的在右边，所以 l 收缩到 mid。

```js
/**
 * @param {number[]} nums
 * @param {number} target
 * @return {number[]}
 */
var lowerbound = function (nums, target) {
  let l = -1,
    r = nums.length;
  //要移动指针缩小范围，就要判断target跟中间值的大小
  //由于初始化两个开区间，那么循环条件就是r - l > 1？？不太懂对不对
  while (l + 1 < r) {
    const mid = Math.floor((l + r) / 2);
    //目标值在中间值左边，收缩右边指针，但是可能有连续的目标值，所以要有等于号
    if (nums[mid] >= target) r = mid;
    //相反
    else l = mid;
  }
  //循环结束之后，l跟r挨着，第一个target是r
  return r;
};
var searchRange = function (nums, target) {
  const start = lowerbound(nums, target);

  const end = lowerbound(nums, target + 0.5) - 1;
  //怎么，判断有没有取到正确答案？记住
  if (start === nums.length || nums[start] !== target) {
    return [-1, -1]; // nums 中没有 target
  }

  return [start, end];
};
```

### 区间内查询数字的频率 -技巧
- 频率考虑Map索引，而且Map值还是个数组！！！

```js

// 构造函数（类似 C++ 的 RangeFreqQuery::RangeFreqQuery）
var RangeFreqQuery = function(arr) {
    // 在 JavaScript 中，this 指向新创建的对象
    // 相当于 C++ 的 this->indexMap
    this.indexMap = new Map();
    
    // 遍历数组，记录每个值的索引
    for (let i = 0; i < arr.length; i++) {
        const val = arr[i];
        if (!this.indexMap.has(val)) {
            this.indexMap.set(val, []);
        }
        this.indexMap.get(val).push(i);
    }
};



// 给原型添加 query 方法（类似 C++ 的成员函数）
RangeFreqQuery.prototype.query = function(left, right, value) {
    // 如果这个值从未出现过
    if (!this.indexMap.has(value)) {
        return 0;
    }
    
    const indices = this.indexMap.get(value);
    
    // 找第一个 >= left 的位置
    const leftIndex = lowerbound(indices, left);
    // 找第一个 > right 的位置（即第一个 >= right+1 的位置）
    const rightIndex = lowerbound(indices, right + 1);
    
    // 范围内索引的个数
    return rightIndex - leftIndex;
};
```

## 开闭区间
### 这些区别的本质是什么？

区间定义决定了边界值的含义：在闭区间中，left 和 right 都是当前可能包含答案的范围；在左闭右开中，right 是第一个不满足条件的位置（或超出范围）；在开区间中，left 是最后一个 < target 的位置，right 是第一个 >= target 的位置。

循环条件随之变化：必须保证区间内至少有一个待检查的元素。

更新规则必须维持循环不变量：这是最核心的。只要你维护好不变量，最终答案就会落在预期的边界上。

mid 落在 target 索引的位置是否受影响？
不，mid 的计算公式 Math.floor((left+right)/2) 在所有实现中都是一样的，只是 left 和 right 的值不同，导致 mid 的取值序列可能不同。但最终搜索到的目标位置是相同的，因为算法逻辑正确。mid 只是一个探针，帮助我们缩小范围，具体落在哪个索引取决于当前区间和数组分布，但最终我们会收敛到正确答案。

### 4. 为什么开区间写法可能更直观？

开区间写法的循环不变量非常清晰：

左边界的左边所有元素 < target（且左边界本身 < target）。

右边界的右边所有元素 >= target（且右边界本身 >= target）。

每次比较 mid 后，根据结果将 mid 赋给左或右边界，区间始终保持开区间，且不变量自动维持。

循环结束时，左右边界相邻，此时右边界就是第一个 >= target 的位置。

这种写法不需要在更新时进行 ±1 操作，不容易出现边界错误，因此许多算法竞赛选手偏爱这种风格。


## 为什么用 floor 而不是 ceil？

### 核心原因：保持区间不变量的正确性

让我们用具体例子说明：

假设数组 `[1, 3, 5, 7, 9]`，`l = -1`, `r = 5`（数组长度）

### 如果用 floor：
```javascript
mid = Math.floor((l + r) / 2) = Math.floor(4 / 2) = 2
// 得到中间偏左的位置，没问题
```

### 如果用 ceil：
```javascript
mid = Math.ceil((l + r) / 2) = Math.ceil(4 / 2) = 2  // 长度偶数时一样

// 但看另一个情况：l = 2, r = 3
mid_floor = Math.floor(5 / 2) = 2  // 指向 l
mid_ceil = Math.ceil(5 / 2) = 3    // 指向 r
```

## 关键问题：当 l 和 r 相邻时

当 `l + 1 = r` 时：
- `(l + r) / 2 = (l + l + 1) / 2 = l + 0.5`
- `floor` 得到 `l`
- `ceil` 得到 `r`

### 如果用 ceil：
```javascript
while (l + 1 < r) {
    mid = Math.ceil((l + r) / 2);  // 当 l+1=r 时，mid = r
    
    if (nums[mid] >= target) {
        r = mid;  // r 不变（还是 r）
    } else {
        l = mid;  // l 变成 r
    }
    // 区间没缩小！可能死循环
}
```

### 用 floor 就安全：
```javascript
while (l + 1 < r) {
    mid = Math.floor((l + r) / 2);  // 当 l+1=r 时，mid = l
    
    if (nums[mid] >= target) {
        r = mid;  // r 变成 l，区间缩小
    } else {
        l = mid;  // l 变成 l（不变？）
    }
    // 检查：如果 nums[l] < target，l 不变，但 r 会变
    // 无论如何，区间一定会缩小
}
```

## 另一个角度：二分查找的本质

我们是在**缩小搜索区间**：
- 用 floor 时，mid 总是偏向左边（较小的一边）
- 这样能保证每次循环都能缩小搜索范围
- 如果 mid 取 ceil，在某些边界情况下可能陷入死循环


## 视觉化理解

数组长度 5（奇数）：`[10, 20, 30, 40, 50]`
下标：`0   1   2   3   4`

```
中间值是下标 2 (30)
用 floor: (0+4)/2 = 2 ✓ 正好是中间
用 ceil:  (0+4)/2 = 2 ✓ 也一样
```

数组长度 6（偶数）：`[10, 20, 30, 40, 50, 60]`
下标：`0   1   2   3   4   5`

```
中间有两个值：下标 2(30) 和 3(40)
用 floor: (0+5)/2 = 2.5 → floor = 2 ✓ 取左边的中间值
用 ceil:  (0+5)/2 = 2.5 → ceil  = 3 ✗ 取右边的中间值
```

## 为什么取左边更自然？

因为**下标从 0 开始**，二分查找的视觉中心自然偏左：
```
[10, 20, 30, 40, 50, 60]
  ↑        ↑        ↑
  l       mid       r
 0        2        5
```
左边有 2 个元素，右边有 3 个元素，左边略少，符合 "floor 取整" 的直觉。

## 数学本质
```javascript
(l + r) / 2 在偶数长度时是 .5
floor 取左边整数
ceil  取右边整数
```

二分查找的标准实现都选 floor（左边），这样：
- 代码简单
- 避免边界死循环
- 符合编程直觉

| 方法 | 是否创建新数组 | 性能 | 返回值 |
|------|---------------|------|--------|
| `push` | 否（修改原数组） | 高 | 新数组长度 |
| 展开运算符 `[...ans, count]` | 是（创建新数组） | 较低 | 新数组 |

对于你的循环场景（m 可能很大），用 `push` 性能更好，因为它不会每次都创建新数组。

## 其他常用数组方法


```javascript
let arr = [1, 2, 3];

arr.push(4);           // 末尾添加 → [1, 2, 3, 4]
arr.pop();             // 末尾删除 → [1, 2, 3]
arr.unshift(0);        // 开头添加 → [0, 1, 2, 3]
arr.shift();           // 开头删除 → [1, 2, 3]
```

所以你可以放心用 `push`，它是 JavaScript 数组的基础方法！

## 上下取整转换函数在珂珂那题的运用

你问的这个问题非常好！作者写的这个求和公式**确实有作用**，但让我解释清楚它到底是什么作用，以及为什么代码里通常直接用 `Math.ceil(p/mid)` 就够了。

### 这个公式的本质是什么？

作者其实是在做**数学上的等价变换**，把 `⌈p/k⌉` 变成了 `⌊(p-1)/k⌋ + 1`。

为什么要这么变？**为了数值计算的稳定性**。

### 举个例子说明区别

假设 p = 7, k = 3：

**方法1（直接上取整）**：
- `Math.ceil(7/3) = Math.ceil(2.333...) = 3`

**方法2（用公式转换）**：
- `⌊(7-1)/3⌋ + 1 = ⌊6/3⌋ + 1 = 2 + 1 = 3`

结果一样，对吧？

### 那为什么要多此一举？

因为**浮点数精度问题**！让我给你看个例子：

```javascript
// 直接使用 Math.ceil
console.log(Math.ceil(5/2));  // 3，没问题

// 但遇到浮点数精度问题呢？
console.log(Math.ceil(10000000000000001/10000000000000000)); 
// 理论上应该是 2，但可能会因为精度问题得到 1
```

`Math.ceil(p/k)` 涉及浮点数除法，在极少数边界情况下可能因为浮点数精度导致错误。

而 `Math.floor((p-1)/k) + 1` 用的是整数运算（或者至少是更稳定的运算），**避免了浮点数精度问题**。

### 代码里到底用哪个？

在实际刷题中，**99%的情况直接用 `Math.ceil(p/mid)` 就够了**，因为：
1. 题目给的数字范围通常不会触发精度问题
2. 代码更简洁易懂

```javascript
// 常见写法（完全没问题）
let sum = 0;
for (const p of piles) {
    sum += Math.ceil(p / mid);
}
```

```javascript
// 作者推荐的写法（避免精度问题）
let sum = 0;
for (const p of piles) {
    sum += Math.floor((p - 1) / mid) + 1;
}
```

### 这个公式怎么推导的？

作者写的这个等式其实是一个数学恒等式：

**⌈a/b⌉ = ⌊(a-1)/b⌋ + 1**（当 a,b 为正整数时）

验证一下：
- a=7, b=3: ⌈7/3⌉ = 3, ⌊6/3⌋+1 = 2+1 = 3 ✓
- a=6, b=3: ⌈6/3⌉ = 2, ⌊5/3⌋+1 = 1+1 = 2 ✓
- a=5, b=3: ⌈5/3⌉ = 2, ⌊4/3⌋+1 = 1+1 = 2 ✓

### 为什么求和公式要写成 n + sum(⌊(p-1)/k⌋) ≤ h？

因为把每个 ⌊(p-1)/k⌋+1 加起来，就是：

sum(⌊(p-1)/k⌋+1) = sum(⌊(p-1)/k⌋) + sum(1) = sum(⌊(p-1)/k⌋) + n ≤ h

这是一种**数学上的简化写法**，对于实际写代码没有太大帮助，但对于理解数学本质还是有意义的。

### 结论

1. **这个公式的作用**：避免浮点数精度问题，用整数运算代替浮点数运算
2. **在刷题中**：直接用 `Math.ceil(p/mid)` 完全没问题
3. **在工程中**：如果对精度要求极高，可以考虑用这个公式

所以，你看不懂这个求和公式完全没关系，它只是一个数学上的等价变换，不影响你理解二分查找的核心思想。直接用 `Math.ceil()` 写出来的代码就能通过 LeetCode 的测试。


## 开闭区间理解

你问到点子上了！二分查找的各种写法确实容易让人混淆。让我用一个清晰的对比帮你彻底理解。

## 核心区别：**区间的定义决定了一切**

不同的写法（开区间、闭区间、左闭右开）本质区别在于**你对区间的理解**，这直接决定了：
1. 左右指针的初始值
2. 循环条件
3. 指针如何移动
4. 最后返回什么

## 三种主流写法对比（以珂珂吃香蕉为例）

### 写法1：左闭右闭区间 [left, right]

```javascript
function minEatingSpeed(piles, h) {
    let left = 1;      // 最小可能速度
    let right = Math.max(...piles);  // 最大可能速度
    
    while (left <= right) {  // [left, right] 不为空
        let mid = Math.floor((left + right) / 2);
        
        if (canFinish(piles, mid, h)) {
            right = mid - 1;  // mid 已经可行，试试更慢的
        } else {
            left = mid + 1;   // mid 不可行，需要更快的
        }
    }
    return left;  // 循环结束时，left 是第一个可行的
}
```

**特点**：
- 初始：left=1（可行？未知），right=max（可行？未知）
- 循环条件：`left <= right` 表示区间还有元素
- 移动规则：排除 mid 后，区间缩小为 `[left, mid-1]` 或 `[mid+1, right]`
- 结束时：left > right，left 就是答案

### 写法2：左闭右开区间 [left, right)

```javascript
function minEatingSpeed(piles, h) {
    let left = 1;      // 最小可能速度
    let right = Math.max(...piles) + 1;  // 注意这里 +1
    
    while (left < right) {  // [left, right) 不为空
        let mid = Math.floor((left + right) / 2);
        
        if (canFinish(piles, mid, h)) {
            right = mid;     // mid 可行，把右边界拉过来
        } else {
            left = mid + 1;  // mid 不可行，排除后往右
        }
    }
    return left;  // 循环结束时，left == right 就是答案
}
```

**特点**：
- 初始：left=1，right=max+1（right 本身不在区间内）
- 循环条件：`left < right` 表示区间还有元素
- 移动规则：right=mid 表示新的右边界（不包含 mid），left=mid+1 表示排除 mid
- 结束时：left == right，这个值就是答案

### 写法3：开区间 (left, right)

```javascript
function minEatingSpeed(piles, h) {
    let left = 0;      // 肯定不可行的
    let right = Math.max(...piles);  // 肯定可行的
    
    while (left + 1 < right) {  // (left, right) 不为空
        let mid = Math.floor((left + right) / 2);
        
        if (canFinish(piles, mid, h)) {
            right = mid;     // mid 可行，更新右边界
        } else {
            left = mid;      // mid 不可行，更新左边界
        }
    }
    return right;  // 循环结束时，right 是第一个可行的
}
```

**特点**：
- 初始：left=0（明确不可行），right=max（明确可行）
- 循环条件：`left + 1 < right` 表示中间还有元素
- 移动规则：直接 left=mid 或 right=mid，不 +1/-1
- 结束时：left 和 right 相邻，right 就是答案

## 关键区别对比表

| 方面 | 闭区间 [L,R] | 左闭右开 [L,R) | 开区间 (L,R) |
|------|-------------|---------------|-------------|
| 左指针初始 | 最小可能值 | 最小可能值 | 明确不可行的值 |
| 右指针初始 | 最大可能值 | 最大可能值+1 | 明确可行的值 |
| 循环条件 | L ≤ R | L < R | L + 1 < R |
| 可行时移动 | R = mid - 1 | R = mid | R = mid |
| 不可行时移动 | L = mid + 1 | L = mid + 1 | L = mid |
| 返回答案 | L | L | R |

## 为什么开区间写法里 left=0 是"明确不可行"？

因为速度 k 必须是正整数，k=0 意味着每小时吃0根，显然吃不完，所以它**肯定不可行**，放在左边界。

而 right=max(piles) 是**肯定可行**的，因为每小时吃最多的一堆那么多，一堆最多1小时吃完，n堆最多n小时，而题目保证 n ≤ h。

## 哪种写法最好？

**没有绝对的好坏**，主要是个人习惯：

- **闭区间**：最直观，但需要处理 ±1，容易出错
- **左闭右开**：STL 标准库风格，很多程序员熟悉
- **开区间**：灵茶山艾府推荐的写法，边界处理最优雅，不用纠结 ±1

我个人觉得**开区间**最容易理解，因为它：
1. 左右边界有明确的含义（不可行 vs 可行）
2. 移动指针时不用想 +1/-1
3. 循环结束时答案直接是 right

希望这个对比能帮你理清思路！
